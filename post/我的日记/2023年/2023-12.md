## Python 为什么不支持 i++ 自增语法，不提供 ++ 操作符？

我发现这些语言的讲解经常让人摸不着头脑，例如：

> 当我们定义`i = 1000` 时，不同语言会作出不同的处理：
>
> - C 之类的语言（写法 `int i = 1000`）会申请一块内存空间，并给它“绑定”一个固定的名称 i，同时写入一个可变的值 1000。在这里，i 的地址以及类型是固定的，而值是可变的（在一定的表示范围内）
> - Python（写法`i = 1000`）也会申请一块内存空间，但是它会“绑定”给数字 1000，即这个 1000 的地址以及类型是固定的（immutable），至于 i，只是一个名称标签贴在 1000 上，自身没有固定的地址和类型
>
> 所以当我们令 i “自增”时（i = i + 1），它们的处理是不同的：
>
> - C 之类的语言先找到 i 的地址上存的数值，然后令它加 1，操作后新的数值就取代了旧的数值
> - Python 的操作过程是把 i 指向的数字加 1，然后把结果绑定到新申请的一块内存空间，再把名称标签 i “贴”到新的数字上。新旧数字可以同时存在，不是取代关系
>
> 打一个不太恰当的比方：C 中的 i 就像一个宿主，数字 1000 寄生在它上面；而 Python 中的 1000 像个宿主，名称 i 寄生在它上面。C 中的 i 与 Python 中的 1000，它们则寄生在底层的内存空间上……

那是因为这些人想强行将object type 过程的原理解释成一个抽象的概念，或者他们自己就没搞懂。

实际上如果自己实现过一个解释器的话，这些很难懂的概念就会变得很清晰。

## 有意思的表达

a school of fish 一池子鱼

ameliorate the performance 没事找事，干嘛不用improve the performance

> 其实两者都可以用，但有一点区别。**Ameliorate** 的意思是“使某事变得更好，但不一定是完美”，而 **improve** 的意思是“使某事变得更好，并接近完美”。因此，如果某事的性能只是需要稍微改进，可以用 **ameliorate**；如果某事的性能需要大幅度改进，可以用 **improve**。
>
> 例如，如果我们要改进一个机器的性能，使其速度更快，可以用 **improve**；如果我们要改进一个软件的性能，使其更稳定，可以用 **ameliorate**。

## multiprocessing instead of threading

> 看到 Python 有 threading 模块，本以为它可以很好地支持高并发，但实际跑起来效果并不好，比单线程更慢。图二中 8 个并发比图一慢了 0.4s。  研究之后才发现，原来 threading 底层实现并没有支持调用多个 CPU，它是一个假的并发模块😅  Python 是采用基于栈的解释器来运行字节码的，通过全局解释器锁（global interpreter lock，GIL）的机制来确保抢占式的多线程环境相互之间不受干扰。简单来说，多个线程是通过排队抢锁的机制来轮流执行的，每个线程在抢到锁之后，都会进行一次 context 切换，这也是“并发”执行更慢的原因。  那为什么大家经常用 Python 进行并发爬虫不觉得慢呢，那是因为在爬虫应用中，通常涉及到的任务主要是网络请求、文件读写等 I/O 操作，而不是 CPU 密集型的计算任务，反而可以在 I/O 等待的过程中发起了更多任务，充分压榨 CPU 的性能。  要让 Python 真正利用起多核 CPU，可以使用 multiprocessing 模块，它会在每个进程内都启动一个 GIL，这样进程之间就不存在抢锁问题了，只不过对内存的开销要大很多😅

其实只是把任务交给系统了，然后让系统背负“**假并发**”的罪名（CPU 的角度）



## 为什么我不再使用Neovim 以及自己配置的vimrc?

曾经，我是一名vim 传教士，上网浏览网页都用vim 快捷键，我曾以为Neovim 集成的现代开发环境是我最终的归宿，但最近几个月，我甚至都没有打开Neovim（因为上次和vim 交换链接错误导致我一直将就着使用没有配置过的裸vim），这是为什么？

Neovim非常漂亮和现代化。它可以替代一些大型复杂项目的IDE，也可以作为一个简单的文本编辑工具。但正如那句老话说的：人不能靠模仿他人获得成功，必须要做你自己。

自从VSCode 出了Remote SSH 和 Live Share 之后 Serverside-Dev 的范式就变得没那么有吸引力了，而Copilot 更是压倒neovim 的最后一根稻草，太上瘾了，用了就回不去。

那么替代Vim呢？Neovim的许多功能依赖于插件，而这些插件又依赖于Node.js和Python，由于它们的发展速度过快，经常会出现不兼容的问题，导致在开发机上出现各种错误。如果不怎么配置的话，为什么不直接使用Vim呢？

我曾经有一个庞大的`.vimrc` 文件，基本把Vim 的自定义键盘映射全都改了（例如常见的保存被直接映射到`s`），但是现在常常要到别人的工作空间里去使用Vim，原本的快捷键都忘得七七八八，很不适应。要知道，这些都是肌肉记忆，很难改正，索性现在开始强逼自己用原生态的键位`ZZ`.

总之，Neovim现在似乎两头不搭，可能只能吸引一些开源爱好者或者对微软持有敌意的用户了吧？

当然，我在VSCode 里也用Vim mode.
