# 工匠精神是怎么丧失的

昨天为了把后端服务硬塞进一台超小台的机器里，做了很多的性能优化，重写了缓存的数据结构、砍掉一堆没用的模块、把常驻进程都翻修了一遍……前前后后搞了十几个 commit，整整折腾了一天，但最后发现完全都白干了。

一开始，跑完 profiling 一看，“吃内存怪兽”是核心缓存的预加载。这就好像你把一个小小的真空压缩袋拆开，里面的棉被瞬间膨胀，把整个房间都塞满了。

我们原本用了 Python 的 `OrderedDict` 来存 L1 缓存数据，写起来是挺爽的，自动排序，但这后来发现这玩意儿对内存极度不友好。每一条记录都挂着对象头、字符串 key，还得叠加上双向链表的开销。结果就是，磁盘上才 30 多 MB 的文件，塞进内存居然直接暴涨到两百多 MB。

刚开始，我把这个缓存基底整个抽换掉。把内部存储从 `OrderedDict + msgpack` 改成 `NumPy structured array + npz`：按复合键分组，核心数值字段用结构化数组连续存储，查询时不再逐条走 Python 对象，而是用 `searchsorted` 做批量查找；原来每条记录都挂一个 source 字符串，也改成 `int8 source_id + string table`，把重复字符串的成本压掉。
同样大约 25 万条记录、5 个分组：

| 指标         | 改前   | 改后   |
| ------------ | ------ | ------ |
| 稳态内存     | 37 MB  | 12 MB  |
| 冷加载耗时   | 3.0 s  | 0.03 s |
| 加载峰值内存 | 375 MB | 26 MB  |
| 磁盘占用     | 32 MB  | 3.3 MB |


接下来就是各种“断舍离”：

比如那个早就没在用的扫描服务，里面居然还包着一整套浏览器引擎，砍掉之后直接省了几百 MB。还有启动流程也是，以前为了在欢迎画面上秀一句“所有工具已加载”，系统会在一开始就把一堆又肥又大的依赖全拉进来。现在全改成了需要再加载。

另外，我也顺手给后台常驻进程加了锁，有时候守护进程会偷偷跑两份。

然后我把 dashboard 的权限也收回了。之前 dashboard 很多读操作自己扫本地文件、自己格式化数据。改完之后，这些查询尽量走已有后端的 RPC——数据查询、session summary、日志、episode detail、触发记录，都让正在运行的后端来回答，dashboard 自己只负责展示，RPC 失败时才退回本地逻辑，别让 dashboard 再偷偷长成第二个后端。

剩下就是些清理后台噪音的琐事。多实例管理刚加上的时候，前端会扫端口、轮询健康状态、重复做实例发现。这些东西单看都不大，但叠在一起就是持续存在的连接、定时器和请求风暴。后面几次提交把这块明显收紧了：

* 后端端口池从 160 个缩到 20 个
* 前端实例发现不再定时反复扫描，改成更克制的触发方式
* 健康检查不再固定每 5 秒轮询，只在窗口聚焦或页面重新可见时触发
* 时间同步链路合并成共享 singleton，多个组件不再各自维护一条轮询

搞定这些，天都黑了。我满心欢喜地跑起来测试，结果猜猜怎么着？

它还是时不时给我爆个 OOM（内存不足）。这就是 1GB 内存的极限，你把所有能抠的都抠干净了，它还是会在你没察觉的时候突然崩溃。

最后我默默掏出手机，扫码支付——好像五块钱吧，把服务器升级成了 2GB。

重新启动。一切顺畅得不得了，内存连一半都没吃到。

前面我还满脑子在感慨什么“工匠精神去哪儿了”，现在想想，真相其实根本没那么文艺。

当年之所以有那么多所谓的极客精神、工匠精神，很大程度上是因为大家真的只有 1GB 内存，花钱也买不到更多。被逼得没办法，只能退而求其次去拼技术，在每一个 MB 里疯狂抠细节。

但现在呢？只要随便掏五块十块，内存轻轻松松就翻倍了。对于我们这种量级的服务，像我今天做的这些花里胡哨的优化，其实根本就没必要。代码结构乱点就乱点呗，平时稍微带点冗余和累赘又怎样？反正随便加点内存，瞎跑也能跑得飞起。

其实开公司难道不也是同样的道理吗？

在公司账上极其宽裕、业绩红火的时候，一口气招个二三十个漂漂亮亮的美女，没啥活干，大家也照样觉得无所谓；办公室里各种昂贵零食敞开供应、团建花钱就像流水一样，根本没人在意这点“性能损耗”。可要是换到刚创业、穷得叮当响的时候呢？恨不得多买一个显示器都要在购物车里权衡半天、算计好久。

说到底，不管是写代码抠内存，还是开公司抓预算，很多时候，“精打细算”并非什么天生的美德，往往只是因为当时我们手里攥着的资源，实在少得可怜罢了。
