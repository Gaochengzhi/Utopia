# 大模型在机器人上的应用——业界与学术动态调研

从系统工程视角看，大模型在机器人上的落地呈现为一个"分层栈"：底层是可实时闭环的控制与安全约束；中层是可执行的技能与策略（模仿学习、强化学习、扩散/自回归策略等）；上层是任务理解、分解与推理（LLM/VLM/MLLM）；旁路是数据与仿真（合成数据、域随机化、数字孪生）以及端侧算力（芯片、量化、编译优化）。2024 年以来，学术与产业的共同趋势是：把"感知—语言—动作"更紧密地统一建模（也就是 VLA / Robot Foundation Models），同时用符号规划、形式化约束、运行时监控等手段补齐可靠性与安全性的缺口。

## 六个最重要的趋势

**趋势 1：VLA 已从"概念验证"进入"开源基线 + 多路线竞争"阶段。**

2024 年的 OpenVLA 把 VLA 做成了一个可复现、可微调、可量化部署的开源基线；随后 π0 / π0.5、RDT-1B、DexVLA、GR00T N1、ChatVLA-2、ABot-M0 等接踵而至，把路线分成了三大类——离散自回归动作、扩散/flow 连续动作、以及更高效的直接连续动作预测。这不再是一两个实验室的探索，而是整个社区在同一条赛道上全面展开。

**趋势 2：数据飞轮正在形成。**

Open X-Embodiment 解决了多实验室、多机器人数据格式统一问题；DROID 把真实家庭/实验环境下的多任务机械臂数据推到 76k demonstrations / 350 小时量级；OpenPI 又把训练规模推到 10k+ 小时机器人数据；NVIDIA 则把 synthetic trajectories 作为增量来源，11 小时就生成 78 万条轨迹，并带来约 40% 性能提升。数据已经不再是"有就行"，而是"谁的数据治理做得更好谁赢"。

**趋势 3：单纯"LLM 直接规划"还远远不够，混合式架构才是正解。**

公开 embodied benchmark 上，最佳通用多模态模型在 EmbodiedBench 上平均只有 28.9%；BEHAVIOR Challenge 2025 的最佳解法也只有约 26% Q-score。这说明：高层推理可以交给大模型，但可执行性、可恢复性、几何可达性仍要靠 PDDL / BT / TAMP / skill library 兜底。

**趋势 4：机器人感知正在从 closed-set 走向 open-vocabulary / 3D grounding / 触觉融合。**

Grounded SAM、Open-Vocabulary SAM 让"任意文本找目标"变得工程可用；RoboSpatial 把空间关系评测推进到 2D+3D；GaussianGrasper、tactile-vision-language 模型则把"语言—几何—接触"联结起来。感知不再只是"看见"，而是"能用语言查询、能定位到 3D、能与接触和操作联动"。

**趋势 5：端侧部署的瓶颈不只在参数量，更在 action decoding、chunking 和 latency 结构。**

OpenVLA-OFT 报告 26× action generation throughput 提升；FAST 报告训练时间最高可降到 5× 更快；Jetson Orin Nano Super、AGX Orin、Jetson Thor 则把机器人端侧算力分成了从 67 TOPS 到 2070 FP4 TFLOPS 的梯度。端侧问题不是"能不能跑 7B 模型"，而是"能不能在有限功耗、有限显存下，以稳定时延跑完感知—推理—动作生成的完整链条"。

**趋势 6：安全性已经从"文本安全"转向"物理安全"。**

AgentSafe、VLA-Arena、EmbodiedBench 都在告诉我们：机器人问题不再只是"会不会胡说"，而是"会不会做出危险动作、不会恢复、不会求助"。与此同时，ISO 10218-1:2025 这类工业标准又把"大模型机器人"拉回到 risk reduction、integration safety 的框架里。2025 年前后安全基准集中爆发，说明社区在"能力提升"之后开始认真转向"可控与可信"。

## 一、机器人专用基础模型与 VLA

### 1.1 这个方向到底在解决什么问题

"机器人专用基础模型"在 2024–2026 年最常见的学术表述是 **Vision-Language-Action（VLA）**：输入包含视觉观测（单/多视角 RGB、RGB-D）、语言指令（或高层语义计划）、机器人本体状态（关节、力觉等），输出为可执行的动作序列（关节空间/末端位姿增量/轨迹块等）。目标是用单一或少数模型覆盖多任务、多场景、甚至多机器人形态。

如果把 RT-1 / RT-2 看作"把 web-scale 视觉语言知识带到机器人"的起点，那么 2024–2026 的主线就是：把这条路线**开源化**（OpenVLA）、**连续控制化**（π0 / RDT / DexVLA）、**推理增强化**（ChatVLA-2 / GR00T N1）、以及**部署高效化**（FAST / OFT / ABot-M0）。

这一方向的瓶颈常被归结为四类：
- **数据**：真实机器人交互数据昂贵且分布碎片化；
- **泛化**：分布外场景、物体与长时序组合任务仍易失败；
- **动作建模**：高频连续控制下，离散动作 token 或低效采样会放大误差与延迟；
- **评测**：不同论文使用不同环境/指标，横向比较困难。

### 1.2 发展脉络

**RT-1 → RT-2**：奠定"VLM 语义 + 机器人动作输出"的基本范式。这两个模型证明了把互联网规模的视觉语言知识带到机器人控制是可行的。

**Open X-Embodiment 与跨本体数据**：汇聚来自 22 种机器人、21 家机构协作采集的数据，展示了 527 种技能、16 万余个任务，给出了跨平台迁移的 RT-X 模型思路。它为后续"用统一数据训练通用 VLA"奠定了基础设施意义上的公共底座。

**OpenVLA（2024，Berkeley / Stanford / TRI / DeepMind 等）**：把 VLA 真正做成了一个**开放、可微调、可量化**的 70 亿参数基线。模型由 Llama 2 语言模型与融合 DINOv2 + SigLIP 特征的视觉编码器组成，在 Open X-Embodiment 的约 97 万段真实演示上预训练。它在 29 个任务上较 RT-2-X（550 亿参数）绝对成功率提升 16.5 个百分点，微调后较 Diffusion Policy 提升 20.4 个百分点。OpenVLA 的意义不只是"又一个模型"——它是第一个让整个社区可以复现、可以微调、可以在消费级 GPU 上跑起来的 VLA。此后几乎所有新工作都在回答同一个问题：**我为什么比 OpenVLA 更好？**

**π0（2024，Physical Intelligence）**：核心贡献是把动作建模从"离散 token 预测"推向 **flow-matching 连续动作生成**。在预训练 VLM 之上学习连续动作分布的向量场，大 VLM backbone 负责高层语义，小 action expert 负责物理控制。这种拆分不是为了"更大"，而是为了更贴近机器人控制的本质需求——连续、平滑、多峰分布。后续 openpi 开源仓库明确提供了 π0、π0-FAST（自回归版本，用 FAST 动作 tokenizer）与 π0.5（更强开放世界泛化）三类模型，并指出基础检查点在超过 10000 小时机器人数据上预训练。

**π0.5 / Knowledge Insulation（2025）**：它引出了一个核心问题——怎样在不破坏 VLM 通用知识的情况下接入动作专家？π0.5 在 VLM 与连续动作专家之间做 knowledge insulation，解决"连续动作专家加入后拖慢训练、破坏知识迁移"的问题。目标是 train fast, run fast, generalize better。openpi 里的工程细节（推理 >8GB 显存，LoRA 微调 >22.5GB）也直接回答了"端侧可用性"的问题。

**OpenVLA-OFT（2025）**：不是新 backbone，而是面向 OpenVLA 的高效微调/推理方案。把 OpenVLA 的 LIBERO 平均成功率从 76.5% 提到 97.1%，同时把 action generation throughput 提升 26×。这个工作非常有启发性——**模型没变，解码和训练流程变了，性能和速度都上去了**。它说明"训练/serving 工程优化"本身就能带来接近"新模型级别"的收益。

**FAST / π0-FAST（2025）**：基于 DCT 的 action tokenization。它说明"动作 token 设计"是 VLA 效率的关键瓶颈；训练时间最高可降到 5× 更快，同时保持对 diffusion-style VLA 的竞争力。可与 π0 结合扩展到 10k 小时规模。

**RDT-1B（2024）**：12 亿参数的 diffusion transformer，面向双臂操作。在 46 个数据集、100 万+ episodes 上训练，代表"扩散式机器人基础模型"路线，强调 few-shot adaptation 与双臂泛化。

**DexVLA（2025）**：带 10 亿参数 diffusion-based action expert 的 VLA，引入 Embodiment Curriculum Learning，试图缓解 VLM 对不同机体的偏置。

**ChatVLA-2（2025，NeurIPS 2025 poster）**：MoE VLA + 两阶段训练。重要点不在"又多了一个 VLA"，而在于**尽量不牺牲 OCR、数学、空间推理**等原始 VLM 能力。作者报告其在 reasoning / comprehension 任务上优于 OpenVLA、DexVLA、pi-zero——但独立复现仍有限。

**GR00T N1（2025，NVIDIA）**：面向人形机器人的开源基础模型，采用 System 2（VLM 慢推理）+ System 1（diffusion transformer 快动作）的分层架构。训练数据混合了真实/仿真机器人轨迹、egocentric human videos 和 synthetic data。产业侧配套 Jetson Thor 与 Isaac/GR00T 工作流，试图形成"数据—仿真—训练—部署"的闭环平台。GitHub 版本（N1.6）进一步说明了架构组合为"VLM + 扩散 Transformer 动作头"，并强调跨形态数据混合训练与后训练适配。

**ABot-M0（2026，阿里高德 AMAP CV Lab）**：试图绕开扩散反复去噪的 latency 问题。它构建了 UniACT 数据集（汇聚 6 个公开数据源，超过 600 万条轨迹、9500 小时交互数据，覆盖 20 余种机器人形态），并提出 Action Manifold Learning（AML）——基于"动作流形假设"，把扩散式学习从预测噪声转为直接预测干净动作序列（a-prediction），以提升解码效率与稳定性。DiT backbone + 语义流/几何流双通道保留多模态信息。它是 2026 年最值得关注的新方向之一，但仍较新，公开复现证据有限。

### 1.4 三条技术路线的比较

2024 年后 VLA 出现了明显分化。从模型类型看：

| 路线 | 代表模型 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- | --- |
| **自回归离散动作** | OpenVLA, π0-FAST | 与 LLM/VLM token 体系天然兼容；LoRA、量化、serving 友好 | 离散化误差、动作序列长、精细连续控制受限 | 工程上最成熟，开源生态最好 |
| **扩散 / Flow 连续动作** | π0、RDT-1B、DexVLA、GR00T N1 | 更自然地表示多峰、连续、chunked 动作；控制表达更强 | 训练和推理通常更慢；端侧部署更难 | 效果潜力更强，但工程更重 |
| **直接连续 clean action** | ABot-M0 | 避免 iterative denoising latency；更适合实时控制 | 较新，复现和 benchmark 证据有限 | 值得关注的 2026 新方向 |

在机器人问题里，从离散到连续的转向有很强的工程动机：连续控制需要更高频率输出、更平滑轨迹和更强的多模态动作分布表达；而标准自回归 token 生成在长序列上更容易累积误差，并且"动作空间离散化设计"本身就是额外瓶颈——如何量化、如何覆盖高维双臂/灵巧手，都不是简单问题。ABot-M0 的项目页说得很直白：传统扩散常学习预测噪声，在控制上效率和稳定性不足，于是提出"动作流形学习"让解码更快更稳。

### 1.5 数据基础设施：为什么 OXE、DROID、UniACT 很关键

**Open X-Embodiment** 的最大价值不在于某个单一数字，而在于**多实验室、多机器人、多任务的统一数据格式与共享训练接口**。它让"跨 embodiment 学习"首次有了基础设施意义。22 种机器人、527 种技能、16 万余个任务——这些数字意味着研究者终于不用各自为战地采数据了。

**DROID（Distributed Robot Interaction Dataset，2024）** 的价值在于"野外分布、多地点采集、可复现硬件栈"。76000 条演示轨迹（约 350 小时），覆盖 564 个场景、86 个任务，由跨洲 50 位采集者在 12 个月内完成，并开源了数据、训练代码和硬件复现指南。这类数据不只是让模型"更大"，而是让模型对场景、操作者、物体的长尾分布更稳——因为它迫使模型面对不同房间、光照、摆放与噪声，而不只是实验室布景。

**UniACT** 的意义在于把多个公开数据集整合到 600 万+ 轨迹、9500 小时量级。这显示 2026 年开始，VLA 的比较对象已经不是"单数据集模型"，而是"谁能把 heterogeneous datasets 整合得更好"。UniACT 的"末端增量 + 旋转向量 + pad-to-dual"方案也体现了社区正在把动作空间治理当成基础设施来做。

### 1.7 技术路线图

**路线 A：把 web / VLM 知识迁移到机器人控制。**

OpenVLA、π0、ChatVLA-2、GR00T N1 的共同目标，都是尽量保留 VLM 的视觉语言知识，同时把它变成可执行的机器人动作。差别在于实现策略——OpenVLA 倾向"把动作也 token 化"；π0 倾向"保留大 backbone，但把动作交给连续 expert"；ChatVLA-2 倾向"通过 MoE / 两阶段训练保留 reasoning"；GR00T N1 倾向"慢推理 + 快动作"的系统分层。

**路线 B：动作头设计决定了训练与部署的上限。**

自回归离散动作利于与 LLM 兼容；扩散/flow 更适合连续控制、多模态动作分布；FAST、ABot-M0 说明下一轮竞争会集中在"如何更快地产生动作"。这不只是学术兴趣——对于真实机器人系统，action decoding 的效率直接决定了控制频率和安全性。

**路线 C：多机体统一。**

DexVLA 的 Embodiment Curriculum Learning、π0.5 的 heterogeneous co-training、OXE 的标准化、GR00T N1 的 humanoid focus，都在回答一个问题：能不能有一个模型跨机械臂、双臂、人形统一？结论是：**正在逼近，但还没统一成功。** 机体几何、控制频率、状态空间、接触动力学差异仍是核心障碍。UniACT 的动作空间统一方案是朝这个方向走的重要一步，但从"格式统一"到"动力学统一"还有很长的路。

### 1.8 商业与产业动态

**Physical Intelligence**：推出 π0、π0.5 和 OpenPI 生态，训练数据达到 10k+ 小时。权威媒体报道其 2024 年融资约 4–6 亿美元，TechCrunch 2026 年 3 月报道其正洽谈约 10 亿美元新融资、估值超过 110 亿美元。openpi 的开源既是技术输出，也是生态构建——既吸引研究者，也形成事实标准。

**NVIDIA**：GTC 发布 Project GR00T（人形机器人通用基础模型研究计划）并配套 Jetson Thor；GR00T N1 / N1.6 作为开源模型层面的实现；同时升级 Isaac 平台，强调从仿真、数据生成到部署的一体化工作流。生态合作/优先用户包括 1X、Agility、Boston Dynamics、Mentee、NEURA 等。

**Figure**：宣布 Series C 超过 10 亿美元 committed capital，估值约 390 亿美元；提出未来四年出货 100,000 台 humanoids 的目标。公司强调从模型到制造的一体化扩张。

**Covariant**：发布 RFM-1，强调让机器人具备更强的"类人推理"与物理反应预测能力，用多模态机器人数据支撑"物理世界模型"。

### 1.9 核心挑战与开放问题

1. **泛化性仍不够**：LIBERO-Plus 这类系统扰动评测说明模型在"相机/背景/布局变化"下仍会显著掉点。跨场景、跨物体、跨机体的"看起来会"与"真实能做"仍有落差。
2. **数据效率不足**：即便 10k 小时看起来很大，和互联网语料相比仍然小得多。DROID 用可复现硬件和分布式采集缓解瓶颈，但"长时任务""高自由度灵巧手""全身人形"仍缺足量开放数据。
3. **部署成本高**：扩散/flow 动作头在 edge deployment 上更吃亏。
4. **多形态统一未解**：数据标准化解决了"格式问题"，还没解决"动力学统一问题"。
5. **安全与恢复机制不足**：长时序真实环境里，失败恢复比单步成功更重要。
6. **统一动作表征与控制接口**：未来需要与控制理论中的约束、稳定性与安全边界更紧耦合（例如安全层、控制屏障函数、可验证规划等）。

## 二、LLM 用于机器人任务规划与推理

### 2.1 核心判断

截至 2026 年初，**LLM 最可靠的角色仍然是"高层规划器/任务解释器"，而不是单独替代 TAMP、技能库和闭环控制器**。真正有效的系统几乎都在走 LLM + PDDL / BT / TAMP / VLA 的混合路线。

用一句话讲清差异：**TAMP 擅长在形式化模型内把事做对；LLM 擅长把"人说的话"变成"形式化模型能接住的东西"，并在失败时给出修复建议。**

### 2.2 标准范式

2024–2026 年研究者普遍采用三段式：

**自然语言指令** → **任务语义解析**（目标、对象、约束、隐含常识） → **任务分解 / subgoal generation** → **符号层表达**（PDDL、BT、任务图、时序逻辑） → **技能调用 / motion planning / VLA 执行** → **执行监控与重规划**

这套范式的重要性在于：大模型擅长把不完整、不规范、含常识的自然语言"翻译"为结构化意图，而真正的可达性、碰撞、安全、恢复，仍要靠传统机器人模块兜底。

一个重要趋势是把传统符号规划与 LLM 结合，以减少"幻觉式计划"：用 LLM 生成/修正 PDDL 问题、再由经典规划器（如 Fast Downward）做启发式搜索，失败则回到 LLM 再生成，构成"生成—验证—重试"的闭环。

### 2.3 代表性工作

| 工作 | 时间 | 主要内容 | 价值 |
| --- | --- | --- | --- |
| **EmbodiedBench** | 2025 (ICML) | 评测 24 个主流多模态大模型；最佳 GPT-4o 平均仅 28.9%；把评测拆成高层（任务分解/规划）与低层（连续控制/空间推理）；把失败分为感知/推理/规划三类 | 最适合说明"通用模型离真正 embodied intelligence 还很远" |
| **PARTNR** | 2025 | 面向人机协作的规划与推理 benchmark；10 万条自然语言任务；含空间、时间、异质体约束 | embodied planning 难点不只是"单机器人做事"，而是多 agent 协调 |
| **LaMMA-P** | 2025 | LM + PDDL + heuristic search；成功率提升 105%、效率提升 36%；配套 MAT-THOR 多智能体长程任务基准 | "LLM + 符号规划"的典范 |
| **FlowPlan** | 2025 | 多阶段零样本 LLM 规划：任务信息检索 → 语言推理 → 符号规划 → 逻辑校验 | CoT 在机器人里不是"写自然语言"，而是显式中间结构 |
| **Few-shot LLM-Planner (ALFRED)** | 2024–2025 | 用 <0.5% paired data 达到近期全数据方法相当表现 | LLM 在 sample efficiency 上很有吸引力 |
| **LLM-GROP** | IJRR 2025 | 视觉 grounding 的 TAMP；真实世界对象重排 84.4% | 把 LLM 放进 TAMP 是为了让任务更贴近自然语言和视觉场景 |
| **LLM-as-BT-Planner** | 2025 | 用 LLM 生成行为树用于机器人装配 | 行为树比自由文本更适合 runtime monitor 和安全约束 |
| **T³ Planner** | 2025 | 用 STL 形式化验证 + LLM 自纠错的运动规划框架 | "LLM 负责生成候选、形式化工具负责验真"的可重复模板 |
| **LLM³** | 2025 | LLM-based task and motion planning framework | 混合规划谱系 |

### 2.4 Chain-of-Thought 在机器人规划里的真实形态

机器人里的 CoT，往往不是"让模型在屏幕上写很多思维步骤"，而是以下几种更工程化的形式：

1. **显式 subgoal list**：先列子目标，再配技能。
2. **PDDL / symbolic state**：把 CoT 变成可检查的逻辑状态转移。
3. **Behavior Tree**：把规划输出变成有 fallback / retry / monitor 机制的可执行树。
4. **STL/LTL 约束**：把时空约束分解为模块，生成候选后用验证器检查并循环修正。
5. **多阶段 pipeline**：先读任务，再推理，再规划，再逻辑校验。

FlowPlan 是这种思路的代表；LaMMA-P 把 LLM 推理嵌进 PDDL 搜索；LLM-as-BT-Planner 把自然语言规划约束成行为树；T³ Planner 则引入时序逻辑验证器做闭环。换句话说：**机器人领域最有价值的 CoT，不是"多想一点"，而是"把中间思路变成可验证的结构"。**

### 2.5 与传统 TAMP 的关系

| 维度 | 传统 TAMP | 纯 LLM 规划 | 混合式（当前最优路线） |
| --- | --- | --- | --- |
| 自然语言理解 | 弱，需人工建模 | 强 | 强 |
| 物理可达性/碰撞约束 | 强 | 弱 | 强 |
| 欠说明任务的常识补全 | 弱 | 强 | 强 |
| 可验证性 | 强 | 弱 | 中到强 |
| 工程落地 | 稳，但开发成本高 | 易原型，难可靠 | 当前最实用 |
| 代表工作 | 经典 TAMP | 纯 prompt planner | LaMMA-P、LLM-GROP、T³ Planner |

结论很清楚：**传统 TAMP 解决"能不能做"，LLM 解决"用户到底想让我做什么"，两者结合才更像真实机器人系统。**

### 2.6 Benchmark 解读：ALFRED、BEHAVIOR、EmbodiedBench

**ALFRED**：仍然是经典长时序 embodied instruction following benchmark。但近期 leaderboard 顶部仍主要被专门化 embodied agents 占据，不是"纯 LLM planner 一统天下"。LLM 的价值更多体现在 few-shot / low-data 规划。并且更复杂、更自然的语言表述会显著增加困难——"语言模型懂语言"不等于"机器人就能执行复杂语言"。

**BEHAVIOR / BEHAVIOR Challenge 2025**：基于 π0.5 路线的方案拿到约 26% 的 Q-score。这不是"成绩低所以方法没用"，恰恰说明：长时序家庭环境 + 部分可观测 + 多物体状态变化的真实难度远高于视觉问答。论文明确指出难点包括 object states 非马尔可夫、缺少 recovery demonstrations、训练流形过窄。

**EmbodiedBench**：最佳模型 GPT-4o 平均 28.9%。论文把失败分成 perception、reasoning、planning 三类，对应流水线的不同阶段（状态描述、反思推理、计划生成）。你可以用它直接解释"为什么 LLM 不能单独端到端承担机器人系统"。这也是讲"failure mode"最好的素材——把机器人失败归因到"看错/想错/计划错/执行错"，分别讨论工程补救。

### 2.7 典型失败模式

**失败模式 1：语义上对，物理上错。** 模型能正确分解"拿起杯子、放到水槽左侧"，但并不知道"左侧"到底是相对于观察者、机器人底座还是水槽正面。这类问题在 EmbodiedBench 的 spatial awareness / visual perception 子集里非常典型。

**失败模式 2：计划看起来合理，但不具可执行性。** 比如生成了顺序正确的 plan，却包含不可达抓取、遮挡对象、缺少前提条件。这就是为什么 LaMMA-P 要把 LLM 输出送进 PDDL + heuristic search，LLM-GROP 要把任务规划与 motion planning 交织。

**失败模式 3：不会恢复。** BEHAVIOR Challenge 论文明确指出，难点包括 object states 非马尔可夫、缺 recovery demonstrations、训练流形过窄。长时序真实任务里，失败恢复比单步成功更重要。

**失败模式 4：幻觉式子任务与对象误绑定。** LLM 生成"看似合理但环境不存在"的对象/工具/地点，导致后续动作无效或危险。这类错误在 EmbodiedBench 中被归入"感知错误导致状态描述错误"或"规划错误导致偏离目标"。

**失败模式 5：长时任务的"雪球效应"。** 前几步小偏差引发后续状态偏离，最终计划整体不可执行；在多步骤家务任务、多人/多机器人协作中尤为明显。大量系统因此引入"反思—纠错—重规划"循环。

**失败模式 6：符号规划接口脆弱。** 把自然语言转换成 PDDL/动作前提时，命名、类型、否定条件、对象集合等细节很容易出错。许多系统因此增加 PDDL 验证器或结构化输出约束。

### 2.8 关于 o1 / o3 类"强推理模型"

一个相对谨慎的结论：推理更强的模型很可能提升高层任务分解、约束整理、异常解释和多步重规划能力。但公开、可核实的 embodied benchmark 文献目前仍主要集中在 GPT-4o、Claude、Gemini 和开源 MLLM；尚未出现足够成熟、可直接引用的 o1 / o3 在机器人 benchmark 上的系统性 peer-reviewed 成绩。最好表述为：**"强推理模型对高层规划很有潜力，但 embodied grounding 仍是瓶颈，证据尚不充分。"**

### 2.9 开放问题

1. **如何把 uncertainty 显式纳入规划？** 现在很多系统会给出 plan，但不会说"我不确定这一步是否可执行"。
2. **如何把规划做成闭环？** 不是一次生成 plan，而是执行—观察—纠偏—再规划。实时性与可部署性是关键——LLM 规划若无法在秒级内给出稳定输出，很难在真实机器人上闭环。
3. **如何让 symbolic checker 覆盖更多真实物理约束？** PDDL 很强，但接触、摩擦、变形、视觉遮挡都很难写全。
4. **如何衡量"计划质量"而非只是任务是否完成？**
5. **多机器人协作与任务分配**：LaMMA-P 强调把 LLM 语义理解与 PDDL 规划结合用于异构多智能体长时任务，但真实世界还需处理通信延迟、观测不完备与安全责任划分。
6. **把物理约束"内化"为推理能力**：当任务涉及摩擦、接触、稳定、碰撞风险时，仅靠语言常识不足，需要显式物理约束或控制屏障类机制介入。

## 三、多模态感知与场景理解

### 3.1 核心判断

机器人感知的难点不是"识别一张图里有什么"，而是要支持**行动**。大模型在机器人感知里不是简单替代传统视觉，而是在做一件更关键的事：**把"看见"升级成"能用语言查询、能定位到 3D、能与接触和操作联动"的开放世界感知系统。** 具体来说，2024 年以来 VLM/MLLM 在机器人中最常见的落地方式包括：开放词汇识别与指代、视觉问答式状态估计、空间推理与导航、以及把感知输出转成结构化表示（对象列表、关系图、可操作区域、3D 场景图）。

### 3.2 四条主线

**主线 A：开放词汇目标识别与分割。**

Grounded SAM 把 Grounding DINO 的文本检测和 SAM 的像素级分割结合起来，让"找红色马克杯""分割桌上最左边那个盒子"这种任务工程上可用（zero-shot SegInW 48.7 mAP）。Open-Vocabulary SAM 进一步实现分割 + 识别一体化，把类别规模推进到约 22000。这两个工作不应被当成"独立终点"，而是 perception stack 的基础件——CLIP / DINO 提供开放语义特征与强视觉表征，Grounded SAM / OVSAM 把语义特征变成可用的像素级目标，机器人系统层再把 mask、box、3D pose、grasp affordance、trajectory optimization 串起来。**大模型感知不是替代几何，而是给几何一个开放词汇入口。**

**主线 B：2D—3D 空间理解。**

RoboSpatial（CVPR 2025）的价值在于，它不是单纯多几张图片，而是把 100 万图像、5k 3D scans、300 万空间关系标注联结起来，直接面向 spatial affordance prediction、spatial relationship prediction、robot manipulation。这说明机器人感知评测正在从"认出来"转向"能不能以操作为目标理解空间关系"。

空间推理相关的评测和增强工作还包括：SpatialEval（NeurIPS）提供文本/视觉/视觉+文本三种输入模式，覆盖地图关系、迷宫导航、网格推理、真实场景空间理解四类任务；SpatialRGPT（NeurIPS）通过数据管线与"插件模块"将深度信息融入视觉编码器；MV-RoboBench（OpenReview）聚焦机器人常见的多摄像头设置，评估 VLM 能否利用多视角输入进行空间推理。

一个需要诚实说的结论：2025 年"空间推理"还没有一个像 ImageNet 那样统一压倒性的排行榜。VLM 的"语义理解强"不等于"几何/空间推理强"，需要面向空间关系的专门数据，以及把深度/多视角/3D 结构显式注入模型。

**主线 C：3DGS / NeRF + 语言查询。**

GaussianGrasper 表明，语言条件下的 3D Gaussian field 可以同时承载几何、语义、抓取候选，适合"给我拿那个写着蓝色标签的瓶子"这类操作。HOV-SG（RSS）提出分层开放词汇 3D 场景图用于语言落地导航，报告在开放词汇语义精度上超过基线，同时将表示大小减少约 75%（更利于端侧存储与检索）。Object-Aware Gaussian Splatting 把 3DGS 用于机器人操作场景，以"对象中心更新"实现对动态场景的实时响应（约 30Hz 更新频率），且可利用预训练基础模型在初始化阶段提取语义、降低策略 rollout 时的大模型推理瓶颈。

**主线 D：触觉进入 foundation model 范式。**

大模型感知过去主要靠 RGB；现在触觉不再是附属模态，而是很多 manipulation 任务里不可替代的信息源。

- **T3（Transferable Tactile Transformers）** 强调跨传感器、跨任务的触觉表征学习，解决触觉传感器异构导致的迁移困难。
- **TVL（Touch-Vision-Language）** 数据集将触觉-视觉配对与语言标注结合，用对比学习训练与 CLIP 对齐的触觉编码器，并微调开源 LLM 做触觉描述——为"把触觉纳入语言空间"提供公开基座。
- **TLA / VTLA** 范式进一步将触觉序列通过语言对齐进入策略生成，代表"接触丰富任务"的大模型路线。VLA 不必止于视觉——逐步融合触觉、力觉、点云，走向更完整的具身多模态。
- **PointLLM** 代表"点云—语言"桥接方向（660k 简单 + 70k 复杂 point-text 指令）。
- Nature Communications 的研究展示触觉与视觉融合架构可实现毫秒级滑移感知，说明"多模态传感 + 决策"在灵巧抓取中具有不可替代的控制价值。

### 3.4 开放问题

1. **open-vocabulary detection ≠ manipulation affordance**：看见"杯子"不等于知道从哪抓。从识别到操控还需要可供性、接触几何与力控策略。
2. **2D mask 到 3D pose 仍不稳定**：特别是遮挡、反光、透明材质下。
3. **空间关系需要视角一致性**：机器人看到的"左边"并不天然等于人类说的"左边"。
4. **触觉数据稀缺**：比图像数据更难规模化。
5. **3D 场景表示与实时性冲突**：NeRF/3DGS 很强，但系统复杂度和实时性仍是门槛。
6. **3D 语义记忆的标准化**：HOV-SG 类 3D 语义图、3DGS/NeRF 表示、以及 VLM 的语言记忆如何统一成可检索、可更新、可端侧部署的"世界模型"，仍缺统一接口与评测。

## 四、具身智能与仿真到现实迁移

### 4.1 核心判断

**具身智能不只是"模型有身体"，而是"模型必须在带物理约束、部分可观测、长时序反馈的环境里闭环决策"。** 因此 2024–2026 的关键，不再是单纯"更强策略"，而是更好的仿真平台、更快的合成数据、更稳的 sim-to-real 过桥机制。

过去的 sim-to-real 重点是**动力学 gap**；现在的 sim-to-real 重点是**动力学 gap + 语义 gap + 任务分布 gap**。也就是说，今天的机器人不是"把仿真里学会的关节控制搬到现实"，而是"把带语言、带常识、带多任务结构的具身策略搬到现实"。

### 4.2 为什么在 2025 年突然更热

原因不是单一论文，而是三股力量汇合：

1. **VLM / LLM 已经能处理高层语义。**
2. **VLA 已经能把语义直接落到动作。**
3. **仿真和合成数据终于足够快，能补足真实机器人数据昂贵的问题。**

GR00T N1 特别能体现这三点：它不只训练在真实机器人数据上，而是混合了 human videos、real robot trajectories、sim trajectories 和 synthetic data；配合 Isaac / Omniverse / Cosmos 生成数据，11 小时就能合成 78 万条轨迹，并带来约 40% 的性能提升。

### 4.3 三个最值得关注的平台

| 平台 | 定位 | 2024–2026 关键点 |
| --- | --- | --- |
| **NVIDIA Isaac Lab** | GPU 加速开源机器人学习框架 | 基于 Isaac Sim，集成执行器模型、多频传感器仿真、数据采集管线、domain randomization 工具；统一 RL / imitation / motion planning 工作流 |
| **Habitat 3.0** (Meta) | 人-机协作具身仿真 | accurate humanoid simulation、human-in-the-loop、家庭环境 HRI；把 embodied AI 从单体导航推进到人机协作 |
| **AI2-THOR / ManipulaTHOR** (AllenAI) | 交互式室内仿真 | 面向交互、导航、操作 benchmark；有 sim-real 对应环境；更适合 benchmark/planning 研究而非最高保真物理引擎 |

### 4.4 合成数据与任务生成：用大模型"造环境、造任务、造示范"

**GenSim2**：用具备多模态与推理能力的 LLM 来生成复杂仿真任务（包括含关节物体的长时任务），并用规划/RL 求解器自动生成示范数据。这代表了"用 LLM 扩展仿真任务空间"的路线。

**MimicGen 与 Isaac Lab Mimic**：从少量人类示范合成大量演示（论文给出从约 200 人类示范生成 5 万演示、覆盖多任务的例子）。其思想已被 Isaac Lab 文档中的数据生成器明确引用并工程化。

**RoboTwin 2.0**：把"结构化域随机化"做成系统工程（杂乱、光照、背景、桌高、语言五轴），并用多模态语言模型与仿真-in-the-loop 细化自动生成任务级执行代码。体现"合成数据 + 语言先验 + 自动任务脚本化"的趋势。

**GR00T 工作流**：NVIDIA 在官方博客中明确描述了通过 GR00T-Teleop、GR00T-Mimic、GR00T-Gen 等工作流生成大量合成轨迹并用于模仿学习训练。

### 4.5 Sim-to-Real 的新配方

**配方 1：Domain Randomization 仍然有效，但不再单独使用。**

Isaac Lab 把 domain randomization 作为基础设施。今天 sim-to-real 的共识是：随机化必须有，但它解决不了高层语义和任务多样性。DORAEMON（ICLR）进一步提出通过最大化训练分布熵来自动塑形动力学分布，试图在无需真实数据的情况下更系统地覆盖未知参数。

**配方 2：大模型先验补语义 gap。**

当策略本身带有 VLM/LLM 先验时，仿真与现实之间不只是视觉外观差异，更重要的是语义结构的一致性。OpenVLA / π0 / GR00T N1 这类模型的重要意义正在于：它们把"互联网语义先验"带到了控制。

**配方 3：Synthetic data 不再是"补充品"，而是第一类数据源。**

GR00T N1 的结果已经很明确：在 humanoid foundation model 里，synthetic trajectories 可以是主力增量来源。这会直接改变 2026 之后机器人数据工程的组织方式。

**配方 4：扩散策略的可控微调。**

DPPO 提出扩散策略优化框架与最佳实践，报告在连续控制与机器人学习任务上相对扩散/非扩散基线都有提升，并强调对 sim-to-real 转移的帮助，是"扩散策略从 BC 走向可优化"的代表。

**配方 5：Benchmark 设计也在"向现实逼近"。**

Habitat 3.0 / PARTNR 更关注人机协作、复杂家庭任务；BEHAVIOR 更强调长时序 household manipulation；VLABench 更强调 VLA 在 spatial / physical law / reasoning 上的压力测试。

### 4.6 开放问题

1. **接触丰富任务仍难仿真**：插接、柔性物体、布料、液体等仍不理想。
2. **人类交互建模不足**：Habitat 3.0 迈出一步，但真实 HRI 远复杂于仿真。
3. **大模型的语义先验和物理先验仍脱节**。
4. **sim-to-real 评价成本高**：缺少统一、低成本、可复现的现实机器人 benchmark。需要像 RoboChallenge 这类统一可复现实验环境来闭环验证"仿真改进是否真的提升真实世界成功率"。
5. **合成数据会不会引入新的偏置？** 这会成为 2026 之后的重要问题。
6. **长尾物理与接触**：摩擦、非刚体、工具使用仍是最顽固差距之一。

## 五、端侧部署与硬件加速

### 5.1 瓶颈画像

机器人端侧的大模型问题，本质上不是"能不能跑一个 70亿/10亿模型"，而是：**能不能在有限功耗、有限显存、有限散热下，以稳定时延跑完感知—推理—动作生成链条。** 这使得动作头设计、量化、LoRA、chunked control、硬件选型和模型本身同等重要。

约束比手机/边缘摄像头更苛刻：操控/避障需要毫秒到几十毫秒级闭环；移动平台或人形机器人电池与散热受限；多传感器（多摄像头、雷达、触觉）带来高带宽与同步压力；部署环境复杂，系统必须可预测并具备降级策略。

### 5.2 主流机器人计算平台对比

> 注意：TOPS 与 FP4 TFLOPS 不是同一口径，这里只做平台级对比。价格因渠道与地区差异大，仅供参考。

| 平台 | 算力 | 内存 | 功耗 | 参考价格 | 定位 |
| --- | --- | --- | --- | --- | --- |
| **Jetson Orin Nano Super** | 67 TOPS | 8GB LPDDR5 | 7–25W | ~$249 / ¥2070 | 轻量 VLM/VLA + 多传感器融合入门 |
| **Jetson AGX Orin** | 最高 275 TOPS | 32/64GB | 15–60W | 渠道价约 ¥19189（开发套件） | 中等规模 perception + planning |
| **Jetson Thor / AGX Thor** | 最高 2070 FP4 TFLOPS | 128GB | 40–130W | 渠道价约 $3499（开发套件） | 人形/大模型时代端侧上限 |
| **Qualcomm RB5 (QRB5165)** | ~15 TOPS | 最高 16GB | 低功耗 | 按渠道询价 | 低功耗 + 多媒体/连接型机器人 |
| **Qualcomm RB3 Gen 2** | 12 TOPS | 6GB LPDDR4x | 高能效 | Core Kit ¥3073 / Vision Kit ¥4473 | 低功耗商业机器人/边缘设备 |
| **Hailo-8** | ~26 TOPS | 片上集成（无需外置 DRAM） | 高能效 | 消费级价格波动 | 感知小模型下沉到协处理器 |

**Jetson Thor 为什么值得单独关注：** 它不只是"更快的 Jetson"，更像 NVIDIA 在机器人端侧的一次重新定义——最高 2070 FP4 TFLOPS、128GB 大内存、面向 humanoid / physical AI、官方同时给出相对 AGX Orin 的 7.5× AI performance 与 3.5× energy efficiency。这意味着 2026 之后一个很现实的问题："机器人端侧到底是跑精简模型，还是直接上更重的端侧硬件？" 答案很可能不是二选一，而是"边压缩模型、边升级端侧平台"。

理解平台选型的方式：把 Jetson Orin 视为"通用机器人端侧基线"，Jetson Thor 视为"人形/大模型时代的端侧上限"，RB5/RB3 代表"低功耗 + 多媒体/连接型机器人平台"，专用 NPU（Hailo/Coral）代表"让感知常开、让主控腾挪"。

### 5.3 模型压缩与部署优化

| 技术 | 代表工作 | 核心结论 | 对机器人意义 |
| --- | --- | --- | --- |
| **量化** | OpenVLA | 支持 int8/int4 量化，不明显损害下游成功率 | VLA 不是只能在大 GPU 上玩 |
| **参数高效微调** | OpenVLA + LoRA | LoRA 只训练约 1.4% 参数，效果接近 full FT | 适合快速适配新任务 |
| **动作 token 高效化** | FAST | 训练最高 5× 更快；扩到 10k 小时 | action tokenization 是 VLA 落地瓶颈 |
| **高吞吐 action generation** | OpenVLA-OFT | 26× throughput | 模型没变，解码变了，性能和速度都上去 |
| **知识隔离/专家接入** | π0.5 / Knowledge Insulation | 解决动作专家拖慢训练/破坏迁移 | 连续动作路线也在补部署短板 |
| **避免迭代去噪** | ABot-M0 | 直接预测 clean continuous action | 针对 diffusion latency 的新尝试 |

### 5.4 端侧延迟：可复现的参考数据

端侧推理延迟很容易被营销口径"漂移"。更适合引用的是 MLPerf Inference Edge 这类第三方基准，以及 NVIDIA Jetson 官方整理的 benchmark 页。

Jetson AGX Orin 上的参考数据（MLPerf v3.1）：ResNet 单流延迟约 0.64ms，BERT 单流约 5.71ms，RetinaNet 单流约 11.67ms。而在 v4.0 的生成式模型条目中，GPT-J 60 亿的单流延迟为万毫秒量级（约 10 秒），Stable Diffusion XL 延迟更高——这直观说明**端侧跑大 LLM/大扩散模型即使能跑也很难满足实时控制，必须采用小模型、分层架构或异步推理**。

更工程化的理解：机器人系统通常不会每个控制周期都完整跑一次大 VLA。更常见的是**低频高层推理 + 高频 skill / chunk execution**。因此真正该看的不是单 token 延迟，而是 chunk latency、控制频率、端到端闭环延迟。如果需要放"xx ms latency"这种硬指标，最好在自己的硬件平台上实测，而不是直接照搬论文。

### 5.5 "端侧小模型（<70亿）"的现状

一个需要诚实表达的结论：公开、可核实的 2025–2026 机器人论文里，轻量化重点更多放在"训练/解码效率"，而不是激进地把模型压到极小参数后仍保留标准 benchmark 优势。现阶段最稳的开源基线仍然是 OpenVLA 70亿；更先进的路线如 π0 / RDT / DexVLA / GR00T N1 在动作表达和泛化上更强，但 edge deployment 更难。openpi 明确给出"推理 >8GB 显存"即可跑部分 π 系列推理——这使得 70 亿以下 VLA 在实验室与工程团队可更快落地。

也就是说：**2025–2026 的"轻量化"更多是系统级轻量化（量化 + LoRA + 高效 action decoding + 异步推理 + 技能缓存），而不是单纯参数级轻量化。**

### 5.6 开放问题

1. **端侧瓶颈不止是参数量，而是动作生成结构。**
2. **量化 + LoRA + 高效 action decoding** 是最现实的近端路线。
3. **Jetson Thor** 代表"直接把 humanoid 级算力搬到边缘端"的产业方向。
4. **可预测性**：量化与编译优化可能引入数值漂移，机器人闭环系统需要"精度—延迟—安全"的联合验证流程。
5. **系统级架构**：端侧大模型更可能以"异步推理 + 技能缓存 + 运行时监控"方式存在，而不是直接进入控制环；统一的软件栈接口仍是工程难题。
6. **未来 2 年最重要的研究点之一，会是"如何让连续动作 VLA 也能端侧实时化"。**

## 六、安全性、可靠性与对齐

### 6.1 核心判断

机器人大模型的安全问题，不再只是"会不会胡说"，而是：**会不会把错误感知、错误推理、错误动作变成真实世界风险。** 幻觉不只是说错话，可能导致碰撞、夹伤、火灾、污染等真实后果。这使得安全研究从 NLP 式 alignment，转向 hazardous instruction、runtime shielding、task verification、标准合规。安全问题必须贯穿：训练前数据治理、训练中对齐与约束、部署时运行时监控、事故后溯源与责任界定。

### 6.2 安全评测与 Benchmark：2025 年集中涌现

| Benchmark | 时间 | 关键内容 | 启示 |
| --- | --- | --- | --- |
| **AgentSafe** | 2025 (ICML) | 首个较全面 embodied VLM agent 安全 benchmark；45 个危险场景、1350 个危险任务、8100 条危险指令；含风险感知指令数据集、对抗场景与多阶段测试 | 安全评测从"正常任务成功率"转向"危险指令下是否安全退让/改写/拒绝" |
| **ASIMOV** | 2025 (Google DeepMind 等) | 生成机器人宪法与语义安全基准；用可扩展数据生成方式构造不良情境 | 评估基础模型作为"机器人脑"的语义安全性 |
| **SafeAgentBench** | 2025 | 安全感知任务规划评测；指出 LLM agent 可能"完美执行危险任务" | 需要专门的安全规划评测 |
| **EMBODYGUARD** | 2025 | 家庭助理场景形式化为自然语言指令 + PDDL + 风险标注 | 评估 LLM 是否理解物理安全约束并生成可执行安全计划 |
| **VLA-Arena** | 2025 | 170 个任务、11 个套件；Safety / Distractor / Extrapolation / Long Horizon 四维度 | VLA 评测把安全性与泛化性一起纳入 |
| **EmbodiedBench** | 2025 | 最佳 28.9%；失败分 perception / reasoning / planning | 不安全往往是整条链路的复合失效 |
| **BEHAVIOR 2025** | 2025 | 最佳约 26% Q-score；object states 非马尔可夫、缺 recovery | 长时序任务的失效恢复是安全核心 |

2025 年前后安全基准集中爆发，说明社区在"能力提升"之后开始转向"可控与可信"，并逐步把安全拆成可测的子能力：危险识别、拒绝执行、替代策略、解释与恢复。

### 6.3 机器人大模型的安全挑战比文本模型多了什么

**幻觉会变成物理风险。** 在文本场景里，幻觉可能只是回答错；在机器人里，幻觉可能导致抓错物体、撞到人、碰翻热水、进入危险区域。AgentSafe 把"危险指令"作为一等公民评测，EMBODYGUARD 则用 PDDL 形式化来检验 LLM 是否理解物理安全约束。

**对齐目标更复杂。** 机器人不只是"拒绝危险指令"，还要能给出安全替代方案、在不确定时请求人类确认、在失败后进入安全状态、遵守空间/力/速度/协作等硬约束。这比纯语言模型的 alignment 难得多，更接近控制理论里的 safety envelope / supervisory control。

**可靠性比单次成功更重要。** BEHAVIOR Challenge 强调的 recovery 缺失，正说明：现实机器人最致命的问题不是"第一次没成功"，而是"失败后不会稳态恢复"。

**安全性和泛化性耦合。** 超出训练分布时，模型更容易产生不安全动作。VLA-Arena 的四维度设计正是在测试这一点。

### 6.4 工程路线一：运行时监控与失败预警

真实部署中，即使策略在离线基准很强，也会遭遇分布外失败。**Sentinel（CoRL）** 把问题表述为：需要在部署时监控学习式生成策略的行为并检测未知失败，以避免负面后果。模仿学习策略在偏离训练分布时易失败，需要 test-time 监控。

一个好的类比：把安全系统比作汽车的 ADAS——不是保证永不出错，而是尽早发现"正在走向失败"，触发降级、刹车或请求人类接管。

### 6.5 工程路线二：安全层 / 形式化约束与大模型结合

**ATACOM 安全层 + 通用策略**："Towards Safe Robot Foundation Models"提出在通用策略之后加安全层，用 ATACOM 构造安全动作空间，从架构层面确保满足安全约束，而无需大量安全示范或针对性微调。这条路线的核心思想是：不改模型，改输出空间。

**LLM + STL/LTL 验证的自纠错规划**：T³ Planner 把时空约束分解为模块，LLM 生成候选轨迹，再用 STL 验证器检查可行性并循环修正。同类思路也体现在 LTL-NL 任务框架中：把自然语言子任务映射为 LTL 公式，再设计计划序列完成任务，从而获得更强的正确性表达。

**可验证的计划结构**：LaMMA-P、LLM-as-BT-Planner、LLM-GROP 都属于"把生成能力约束进可验证结构"的尝试。它不是完全 formal verification，但已经是当前最有效的工程替代。

严格意义上的"开放式 VLM/VLA 形式化验证"在公开文献里仍然稀缺。当前更可行的路线是：LLM 负责生成候选计划 → PDDL / BT / TAMP / 几何规划器负责验证与约束 → runtime monitor / shield 负责执行期拦截。

### 6.6 监管与标准化动态

**ISO 10218-1:2025**：工业机器人安全要求标准，强调 inherently safe design、protective measures、information for use，把机器人视为不完整机器的一部分。即便用最先进的大模型，最终落地仍要回到风险评估、系统集成、安全边界的工业标准框架。

**ISO/TS 15066:2016**：协作机器人安全要求技术规范，补充 ISO 10218-1/-2 对协作操作的要求。

**ISO/IEC TR 5469:2024**：直接面向"AI 与功能安全"，描述 AI 用于安全相关功能时的风险因素、方法与流程。这是把"AI 系统的不可预测性"纳入功能安全讨论的关键文献。同时 ISO/IEC CD TS 22440-1 处于"功能安全与 AI 系统要求"的标准开发阶段，说明标准化正在从技术报告走向更硬的要求文本。

**IEEE 人形标准框架**：IEEE Robotics & Automation Magazine 2025 发布了"Robot Ethics: Ethical, Legal, and User Perspectives…"特刊，以及人形机器人标准框架报告。IEEE 当前更多推动"伦理、法律、用户视角"的框架化讨论。

总结：**ISO 更接近"工程安全要求"，IEEE 当前更多推动"伦理与法律框架"。标准化正在从"工业机器人/协作机器人"扩展到"AI 驱动的机器人系统"，核心矛盾是如何把学习系统的统计保证与安全工程中的确定性/可验证性对齐。**

### 6.7 开放问题

1. **如何让模型表达"不确定"？** 不只是给出计划，而是同时给出置信度和风险评估。
2. **如何把 human override 设计进 VLA / planner？** 真实场景中人类接管必须是一等公民。
3. **如何把安全目标直接纳入训练，而不是只靠后处理 shield？**
4. **如何建立统一的真实机器人安全 benchmark？** 当前大量安全基准仍在仿真沙盒中评估，如何在不引入伦理风险的前提下建立真实测试规范。
5. **如何让标准制定跟上 foundation model 的迭代速度？**
6. **责任链与审计**：当系统由 VLM/LLM、规划器、策略模型、控制器、监控器与硬件共同组成，故障归因与审计需要新的系统工程方法与标准接口。
7. **危险指令下的安全退让不足**：感知错误会放大为规划错误，再放大为执行错误——这是整条链路的复合失效。

这几项会决定未来 3–5 年"机器人大模型能否真正走出 demo"。
